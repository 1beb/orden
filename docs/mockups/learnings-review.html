+
+Rendering is host-owned, agent-driven. The agent edits
+`.qmd`/`.md` source, calls `doc_render({path})` (host runs
+quarto, returns `{ok, outputPath, errors}`), verifies `ok`,
+then `panel_open(outputPath)`. Two tools on purpose: the
+verify-then-open step stays an explicit gate.
Why this

This session designed intelligent automation for orden. It settled that quarto/render always runs on the host, but the agent triggers it, confirms completion, then surfaces the result — so a new doc_render MCP tool returns build status and stays separate from panel_open, keeping the verify-then-open gate explicit.

That two-tool contract isn't written down anywhere an agent would read it. Future sessions will guess at one combined tool.
