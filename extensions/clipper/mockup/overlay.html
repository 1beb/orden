Streamable HTTP for stateful agent sessions

M. Rivera · updated 2026-05-30 · 11 min read

A streamable HTTP transport lets a client open a long-lived logical session over ordinary request/response HTTP without holding a socket open the whole time. The server mints a session id on the first initialize call and the client echoes it back on every subsequent request. Intermediaries see plain POSTs, so it survives proxies that would kill a WebSocket.

Session lifecycle

The first request carries no session id. The server generates one, returns it in a header, and keeps the associated transport in memory. If the process restarts, that in-memory map is lost and the next request with a stale id must be answered with a 404 — the client's signal to transparently re-initialize.

Clients should treat a dropped session as recoverable rather than fatal, re-sending the initialize and replaying any unacknowledged work. This keeps long-running agents resilient across server redeploys.

Why not a socket

WebSockets give you push but cost you reconnection complexity and proxy fragility. For a tool that fires occasional, user-initiated bundles, a single POST is both simpler and more robust.

A service worker that is killed between events cannot hold a socket anyway, so the request/response model is not a compromise here — it is the correct fit for ephemeral, event-driven clients.
